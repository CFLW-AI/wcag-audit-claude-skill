Executive Summary

This page has significant accessibility failures across keyboard navigation, focus management, and color contrast. The most critical issues are:

  • Global focus removal (*:focus { outline: none; }) without replacement indicators
  • Click-only UI components that cannot be accessed via keyboard
  • Keyboard trap in modal — focus cannot escape
  • Sticky header obscuring focus indicators
  • Non-compliant focus indicator contrast (1.32:1 vs required 3:1)
  • Multiple color contrast failures in UI borders and text

Detailed Failure Analysis

Failure 1: Global Focus Indicator Removal (SC 2.4.7, SC 1.4.11)

Location: *:focus { outline: none; } (line 24)

Issue: All interactive elements have their browser-default focus outline removed globally without any visual replacement. This makes keyboard navigation completely inaccessible.

Why it fails: SC 2.4.7 requires every interactive element to have a visible focus indicator. The global outline: none removes this entirely, leaving keyboard users with no feedback about which element has focus.

Impact: Critical. Keyboard users cannot navigate the page.

Failure 2: Click-only Cards (SC 2.1.1, SC 4.1.2)

Location: .card-link divs (lines 106-117)

Code:

<div class="card-link" onclick="location.href='/project/1'">

Issue: Cards are implemented as <div> elements with onclick handlers. They are not keyboard accessible because:

  • No tabindex attribute — cannot receive focus via Tab
  • No keydown or keypress` listeners — cannot activate with Enter or Space
  • No semantic role — assistive tech doesn't know they're interactive

Why it fails: SC 2.1.1 requires all functionality to be keyboard accessible. SC 4.1.2 requires proper name, role, and value for custom components.

Impact: Critical. Cannot navigate to projects via keyboard.

Failure 3: Modal Keyboard Trap (SC 2.1.2, SC 4.1.2)

Location: #modal-trap (lines 148-155)

Issue: Modal dialog opened by "Open Settings" button has no escape mechanism:

  • No Escape key handler to close the modal
  • No focus trap to keep focus within the modal
  • Focus can circulate forever within the modal's inputs and button, with no way to exit
  • No role="dialog", aria-modal="true", or aria-labelledby

Why it fails: SC 2.1.2 explicitly forbids keyboard traps. Users must be able to exit using standard keys (Tab, Escape).

Impact: Critical. Keyboard users cannot close the modal.

Failure 4: Focus Obscured by Sticky Header (SC 2.4.11)

Location: .sticky-header (lines 26-31), z-index: 100

Issue: The sticky navigation header is fixed to the top with a high z-index (100). When keyboard users Tab through content, the focus indicator on items beneath the header is completely hidden behind the opaque header.

Why it fails: SC 2.4.11 (new in WCAG 2.2) requires that focused elements are not entirely obscured by other content. A keyboard user has no visual feedback on which element is focused.

Impact: Critical. No focus visibility for content beneath the header.

Failure 5: Non-compliant Focus Indicator Contrast (SC 1.4.11, SC 2.4.7)

Location: .subtle-focus:focus (line 65)

Code:

outline: 1px solid #e0e0e0; outline-offset: 2px;

Contrast ratio: #e0e0e0 on #ffffff = 1.32:1 (required: 3.0:1)

Issue: The focus outline color (#e0e0e0, a very light gray) has such poor contrast against the white background that it is nearly invisible. The 1px width makes it even harder to see.

Why it fails: SC 1.4.11 requires focus indicators to have at least 3:1 contrast against adjacent colors. SC 2.4.7 requires focus indicators to be "clearly visible." A barely-visible outline fails both.

Impact: High. Focus indicator is effectively invisible.

Failure 6: Disrupted Tab Order (SC 2.4.3)

Location: Search input (line 98)

Code:

<input type="search" tabindex="5">

Issue: The search input has tabindex="5", which causes it to jump ahead in the tab order. A positive tabindex value disrupts the natural (DOM-based) reading order.

Why it fails: SC 2.4.3 requires focus order to follow a logical, predictable sequence. Positive tabindex values override DOM order and confuse keyboard users.

Impact: Medium. Tab navigation is non-intuitive.

Failure 7: Mouse-only Expand Control (SC 2.1.1, SC 1.4.13)

Location: .expand-btn (lines 137-143)

Code:

onmouseover="...display='block'" onmouseout="...display='none'"

Issue: Content is revealed on hover only via JavaScript. Keyboard users cannot:

  • Trigger the reveal with focus or keyboard events
  • Dismiss the content via Escape
  • Interact with the revealed content

Why it fails: SC 2.1.1 requires all functionality to be keyboard accessible. SC 1.4.13 requires hover/focus-triggered content to be dismissable and persistently visible if needed.

Impact: High. Content is hidden from keyboard users.

Failure 8: Missing Custom Control ARIA (SC 4.1.2)

Location: .custom-dropdown (lines 122-130)

Issue: Custom dropdown has no ARIA attributes:

  • No role="listbox" or role="combobox"
  • No aria-expanded="true|false" to indicate open state
  • No aria-haspopup="listbox"
  • Options have no role="option" or aria-selected
  • No keyboard event listeners for arrow keys to navigate options

Why it fails: SC 4.1.2 requires custom interactive components to expose name, role, and state to assistive technology. Without ARIA, screen readers have no way to announce the dropdown's purpose or state.

Impact: Critical. Screen reader users cannot understand or operate the dropdown.

Remediation Priority

  1. Critical: Restore visible focus indicators globally. Replace *:focus { outline: none; } with a robust :focus-visible style with at least 3:1 contrast and 2px width.
  2. Critical: Convert .card-link divs to <a> elements or add role="button", tabindex="0", and keydown listeners.
  3. Critical: Add Escape and focus-trap handling to modal. Implement role="dialog" and focus management.
  4. Critical: Reduce sticky header z-index or add focus-visible scrolling logic to prevent obscuring focus.
  5. High: Fix all color contrast failures (search border, footer text, footer border, subtle focus).
  6. High: Add ARIA attributes to custom dropdown and expand button.
  7. Medium: Remove custom tabindex="5" on search input to restore natural tab order.

Methodology

Audit Scope

This audit evaluates HTML source code against all 28 WCAG 2.2 Level AA success criteria. The page was tested in its light-theme default state.

Color Contrast Testing

All foreground/background color pairs were extracted from CSS and verified using the WCAG 2.2 relative luminance algorithm. Contrast ratios were computed programmatically:

Keyboard Navigation Testing

Code was reviewed for keyboard accessibility patterns:

  • Presence of tabindex on non-semantic interactive elements
  • Event listeners for keydown, keypress, and keyup
  • Focus traps and Escape key handlers in modals
  • Focus indicators via :focus or :focus-visible

ARIA Compliance

Custom interactive components were checked for:

  • role attributes describing the component type
  • aria-label or aria-labelledby for accessible names
  • State attributes: aria-expanded, aria-selected, aria-checked
  • Relationship attributes: aria-haspopup, aria-owns
